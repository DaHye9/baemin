> 아래 그림은 Applied Software Measurement: Global Analysis of Productivity and Quality 의 소프트웨어 결함
생산, 발견, 수정 비용에 대한 그래프입니다.
결함 수정 비용을 감소시키기 위하여 필요한 활동을 "간략히" 작성해주세요.

앞단계에서 모든 결함을 발견할 수 있다면 좋겠지만 서비스 운영 구조 상 불가능합니다. 하지만 릴리즈 이후에 소프트웨어 결함이 발견되는 경우 소프트웨어 결함을 미리 발견하지 못한 나비효과가 기하급수적으로 커지기 때문에 릴리즈 과정 이전에 최대한 정확하고 많은 테스트가 이루어 져야 합니다.

### 코딩 단계 / 유닛 테스트
* 유닛 테스트의 커버리지를 최대한 높여야 합니다. 코드 오너가 유닛 테스트를 작성한다면 코드 오너의 주관이 개입될 수 있기 때문에 다른 개발자와 코드 리뷰로 크로스 체킹을 꼼꼼하게 하면 좋습니다.
* CI/CD와 연동하여 유닛 테스트뿐만 아니라 type checking, sonarQube와 같은 타입 및 잠재적인 버그 가능성을 코드 머지 전에 모두 테스트합니다.

### Functional Test
기능에 대한 정확한 이해를 바탕으로 기본적인 테스트 케이스뿐만 아니라 엣지 케이스들까지 잘 설계하여 테스트를 수행해야 합니다. 이 때 핵심 기능에 더 정교한 테스트를 할당하여 테스트 효율성도 최대한 높이면 좋습니다.

### System Test
시스템 테스트 단계 부터는 실제 운영 환경에서의 테스트가 들어가기 때문에 서버 문제, 스케일 테스트, 보안 이슈와 같은 functional test에서 테스트하지 못하는 부분 위주로 테스트해야 합니다. 이 단계에서 문제가 발생하면 아주 간헐적인 이슈더라도 릴리즈 이후에는 문제가 더 빈번하게 발생할 가능성이 있으므로 root cause analysis 및 해결을 완벽하게 해야 합니다.
